# agent-mem v0.4 Spec

## Goal

Turn `agent-mem` into a CLI-first memory system for coding sessions where the user can connect an Obsidian vault during setup and have session memory saved directly as Obsidian-compatible Markdown notes.

The product should reduce context usage automatically, produce structured memory artifacts, and make those artifacts useful both inside Obsidian and outside it.

## Product Position

`agent-mem` is not primarily an MCP server.

`agent-mem` is a local memory runtime that:

- captures coding-session context
- compresses that context into structured summaries
- saves it in Obsidian-native Markdown when available
- falls back to local Markdown when Obsidian is not configured
- returns compact recall for future work

IDE and MCP integration are delivery mechanisms, not the core product.

## Primary User Story

The user installs the package and runs:

```bash
agent-mem init
```

During setup, the CLI should ask:

- do you want to connect an Obsidian vault?
- if yes, what is the path to the vault?
- if no, use local project storage

If the user connects Obsidian, `agent-mem` should write notes in Obsidian-friendly Markdown automatically.

This means:

- standard `.md` notes
- YAML frontmatter
- wiki-links like `[[Project Name]]`, `[[Decision - Auth]]`, `[[File - app.py]]`
- folder placement inside a predictable Obsidian path

## Core Requirements

### 1. Obsidian-First Storage

When Obsidian is enabled, the primary store is the user's vault.

Expected default path:

```text
<vault>/Memory/Agent-Mem/
```

Expected note types:

- one session note per summarization event
- one rolling project index note
- optional decision notes in a later phase

### 2. Obsidian-Compatible Markdown Format

Every saved session note must be valid plain Markdown and also useful inside Obsidian.

Minimum format:

```md
---
title: "Session - agent-mem - 2026-04-08 10:30"
project: "agent-mem"
type: "agent-session"
created: "2026-04-08T10:30:00+05:30"
tags:
  - agent-mem
  - coding-session
  - memory
---

# Session: [[agent-mem]]

## Summary

...

## Decisions

- [[Decision - CLI First]]: Keep core workflow CLI-first.

## Files

- [[File - src/agent_mem/memory.py]]

## Open Tasks

- ...

## Next Step

- ...
```

Required Obsidian features:

- YAML frontmatter for metadata
- wiki-links for graph connections
- stable note naming
- plain Markdown compatibility outside Obsidian

### 3. Local Fallback Storage

If the user skips Obsidian during `init`, the system must still work.

Fallback path:

```text
<project>/.agent-memory/
```

Fallback artifacts:

- `memory.md` as rolling memory log
- optional `sessions/` folder in a later phase if structure becomes necessary

Fallback mode must not be treated as a second-class broken mode. It should be a complete, reliable baseline experience.

### 4. CLI-Driven Context Compression

The product must reduce reliance on IDE instructions and long custom prompt files.

The CLI should own summarization behavior.

Planned commands:

```bash
agent-mem init
agent-mem summarize
agent-mem recall "<query>"
agent-mem status
agent-mem watch
```

Command meanings:

- `init`: connect Obsidian or configure fallback storage
- `summarize`: save a structured session summary immediately
- `recall`: return the most relevant memory for a task or query
- `status`: show active storage mode and paths
- `watch`: support future auto-summarization triggers

## Desired v0.4 Workflow

### Setup

```bash
pip install <package>
agent-mem init
```

During init:

1. ask whether to use Obsidian
2. if yes, collect and validate vault path
3. if no, configure local fallback
4. create required folders and starter files
5. print the active memory mode clearly

### Saving Memory

User or host system runs:

```bash
agent-mem summarize
```

The CLI should:

1. accept summary input directly, from stdin, or from a file in a later phase
2. normalize the summary into a structured note shape
3. write the note to Obsidian or local fallback
4. update project-level index files
5. print the save location

### Recalling Memory

User or host system runs:

```bash
agent-mem recall "auth flow"
```

The CLI should return:

- current active decisions
- relevant recent sessions
- open tasks
- next-step context

It should not dump entire notes unless explicitly requested.

## Automatic Context Reduction

The product needs better automation than "tell the IDE to summarize when it feels long."

v0.4 should define the automation model, even if implementation is incremental.

Trigger ideas:

- manual `agent-mem summarize`
- turn-count threshold
- transcript size threshold
- elapsed-time threshold
- git event hooks in a later phase

Expected output of automatic summarization:

- session note written
- project index updated
- compact handoff generated for the next session

## Storage Model

### Obsidian Mode

Folder layout:

```text
<vault>/
  Memory/
    Agent-Mem/
      Index.md
      agent-mem-2026-04-08_10-30-session.md
      agent-mem-2026-04-08_13-45-session.md
```

`Index.md` should include:

- project overview
- links to recent sessions
- active decisions
- active open tasks

### Fallback Mode

Folder layout:

```text
<project>/
  .agent-memory/
    memory.md
```

Later enhancement:

```text
<project>/
  .agent-memory/
    memory.md
    sessions/
```

## Non-Goals For v0.4

- building a complex vector database
- making MCP mandatory
- provider-specific LLM orchestration
- deep IDE-specific automation

Those can come later. The immediate goal is reliable structured memory and Obsidian integration.

## Success Criteria

v0.4 is successful if:

- a user can connect Obsidian during CLI setup
- `agent-mem` writes valid Obsidian-friendly Markdown notes directly into the vault
- the notes create visible graph connections through wiki-links
- fallback mode still works with no Obsidian installed
- summarization and recall are CLI-driven and usable without MCP
